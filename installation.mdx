title: 'Installation'
description: 'Add tract to your Rust project, install the Python package, or set up the CLI tool.'
---

<Tabs>
  <Tab title="Rust">
    Add tract to your `Cargo.toml`. The `tract` crate is the recommended entry point — it re-exports the ONNX, NNEF, and pulse functionality with a unified API.

    <Steps>
      <Step title="Add the basic dependency">
        For ONNX inference, add the `tract` crate (the unified public facade introduced in 0.23):

        ```toml Cargo.toml
        [dependencies]
        tract = "0.23"
        anyhow = "1"
        ```

        <Note>
          The `tract` umbrella crate and the `into_model()` API were introduced in **0.23**. If you are using the previous stable release (0.21.x), the crate was called `tract-rs` and the method was `into_tract()`. All examples in these docs target the 0.23 API.
        </Note>

        This pulls in `tract-onnx`, `tract-nnef`, `tract-core`, `tract-linalg`, and `tract-data` transitively.
      </Step>

      <Step title="Choose additional features">
        Use individual crates if you need only a subset of functionality. For embedded targets where startup time matters, you can depend on `tract-nnef` directly and skip the ONNX parser entirely.

        <CodeGroup>

        ```toml ONNX only
        [dependencies]
        tract = "0.23"
        ```

        ```toml NNEF only (embedded)
        [dependencies]
        tract-nnef = "0.23"
        ```

        ```toml TensorFlow 1.x
        [dependencies]
        tract-tensorflow = "0.23"
        ```

        </CodeGroup>
      </Step>

      <Step title="Enable GPU acceleration (optional)">
        GPU support is available on Linux and Windows through the `tract-cuda` crate, and on Apple platforms through `tract-metal`. These are included automatically when you use the `tract` umbrella crate on supported targets.

        For Linux or Windows with CUDA:

        ```toml Cargo.toml
        [dependencies]
        tract-cuda = "0.23"
        ```

        <Note>
          GPU crates are activated at runtime based on the target OS. On `linux` and `windows` targets, `tract-cuda` is linked automatically when present. On Apple platforms, `tract-metal` is used instead.
        </Note>
      </Step>
    </Steps>

    <Note>
      **MSRV**: tract requires Rust **1.89.0** or later. Ensure your toolchain is up to date:

      ```bash
      rustup update stable
      rustc --version
      ```
    </Note>
  </Tab>

  <Tab title="Python">
    The `tract` Python package wraps the native Rust library through C FFI and is distributed as a pre-built wheel on PyPI.

    <Steps>
      <Step title="Install from PyPI">
        ```bash
        pip install tract
        ```

        Pre-built wheels are available for Linux (x86\_64, aarch64) and macOS (x86\_64, arm64). Windows support is experimental.
      </Step>

      <Step title="Verify the installation">
        ```python
        import tract
        print(tract.version())
        ```

        This prints the version string of the underlying native library.
      </Step>

      <Step title="Install with numpy (recommended)">
        tract accepts and returns `numpy` arrays directly. Install it alongside tract:

        ```bash
        pip install tract numpy
        ```
      </Step>
    </Steps>

    <Tip>
      Use a virtual environment to avoid dependency conflicts:

      ```bash
      python -m venv .venv
      source .venv/bin/activate
      pip install tract numpy
      ```
    </Tip>
  </Tab>

  <Tab title="C / FFI">
    tract exposes a stable C API through the `tract-ffi` crate. This enables integration from C, C++, or any language that can call C functions via FFI.

    <Steps>
      <Step title="Add the tract-ffi crate">
        ```toml Cargo.toml
        [dependencies]
        tract-ffi = "0.23"
        ```

        Building this crate produces a shared library (`libtract_ffi.so` on Linux, `libtract_ffi.dylib` on macOS, `tract_ffi.dll` on Windows).
      </Step>

      <Step title="Use the C header">
        The C header file is generated alongside the shared library. Include it in your C project:

        ```c
        #include "tract.h"
        ```

        The header is published with each release in the [GitHub releases](https://github.com/sonos/tract/releases/latest) page.
      </Step>
    </Steps>

    <Info>
      The Python bindings (`pip install tract`) are themselves built on top of `tract-ffi`. Reading the Python source in `api/py/tract/` is a useful reference for C FFI usage patterns.
    </Info>
  </Tab>

  <Tab title="CLI">
    The `tract` command-line tool is useful for inspecting, profiling, benching, and converting models without writing any code.

    <Steps>
      <Step title="Install with Cargo">
        ```bash
        cargo install tract
        ```

        This compiles the latest release from source. Requires Rust 1.89.0+.
      </Step>

      <Step title="Or download a prebuilt binary">
        Prebuilt binaries for macOS (Intel), Linux (Intel), ARMv7, and ARMv8 are available on the [GitHub releases page](https://github.com/sonos/tract/releases/latest).

        ```bash
        # Example: download the Linux x86_64 binary
        wget https://github.com/sonos/tract/releases/latest/download/tract-x86_64-unknown-linux-gnu.tar.gz
        tar xzf tract-x86_64-unknown-linux-gnu.tar.gz
        ./tract --help
        ```
      </Step>

      <Step title="Verify the installation">
        ```bash
        tract --version
        ```

        Basic usage — load a model and dump its IR:

        ```bash
        tract mobilenetv2-7.onnx -i 1,3,224,224,f32 dump
        ```
      </Step>
    </Steps>

    <Tip>
      Use the `-O` flag to see the fully optimized form of the graph, and `bench` to measure throughput:

      ```bash
      tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 bench
      ```
    </Tip>
  </Tab>
</Tabs>

## Version compatibility

| Component | Minimum version |
|---|---|
| Rust toolchain | 1.89.0 |
| tract crate | 0.23 |
| Python | 3.8+ |

<Note>
tract follows a single-version policy: all `tract-*` crates in a project must use the same version. Mixed versions will cause linker errors or API mismatches.
</Note>
