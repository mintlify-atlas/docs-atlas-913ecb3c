## dump

Display the model graph in tract-OPL intermediate representation. This is the default subcommand when none is specified.

The displayed form is *decluttered*: training artefacts are stripped and the graph is simplified as much as possible while remaining platform-independent. It can be serialized to NNEF. This is not the same as the *optimized* form, which is tuned for a specific CPU.

```bash
# Load, analyze, and dump the decluttered graph
tract mobilenetv2-7.onnx -i 1,3,224,224,f32 dump

# Dump the optimized graph
tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 dump

# Dump with cost annotations (FLOPs per operator)
tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 dump --cost

# Dump with profiling data (running time per operator)
tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 dump --profile --cost
```

Use `--pass load` or `--pass analyse` to inspect earlier pipeline stages when input types are unknown:

```bash
tract mobilenetv2-7.onnx --pass load dump
tract mobilenetv2-7.onnx -i 1,3,224,224,f32 --pass analyse dump
```

### dump options

| Flag | Description |
|---|---|
| `--cost` | Annotate each node with arithmetic cost (FLOPs) |
| `--profile` | Run the model and annotate each node with measured time |
| `--json` | Emit performance info as JSON |
| `--audit-json` | Emit the full model graph as JSON for machine consumption |
| `-s / --summary` | Show a short summary: properties, inputs, and outputs |
| `--const` | Also display constant nodes |
| `--info` | Show operator inner information |
| `--io-long` | Show full input/output tensor information |
| `--axes` | Compute and display axis tracking across the graph |
| `--nnef <path>` | Serialize the graph to a `.nnef.tgz` file |
| `--nnef-dir <path>` | Serialize the graph to a NNEF directory |
| `--tflite <path>` | Serialize the graph to a TFLite flatbuffer |
| `--inner <name>` | Navigate into a named sub-model |

## bench

Benchmark inference latency using tract's built-in timing harness. Always pass `-O` when benchmarking so you measure the optimized execution path.

```bash
tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 bench
```

<Tip>
  If you omit `-O`, tract will warn: "Profiling a non-optimized model." The benchmark will still run but results won't reflect production performance.
</Tip>

### bench options

| Flag | Description |
|---|---|
| `--max-loops <N>` | Maximum number of iterations (default: 100,000) |
| `--max-time <ms>` | Maximum total run time in milliseconds (default: 5,000) |
| `--warmup-loops <N>` | Number of warm-up iterations before timing starts |
| `--warmup-time <ms>` | Warm-up duration in milliseconds |

## criterion

Benchmark using the [`criterion`](https://docs.rs/criterion) crate, which provides statistically rigorous measurements including confidence intervals and outlier detection.

```bash
tract -O mobilenetv2-7.onnx -i 1,3,224,224,f32 criterion
```

<Note>
  `criterion` generates an HTML report in `target/criterion/` when run from a Cargo workspace. When invoked as a standalone binary the report is written to the current directory.
</Note>

## run

Run the model on actual input data. Input can be provided via command-line flags, an `.npz` bundle, or randomly generated.

<CodeGroup>

```bash Run with npz input
tract model.onnx -O run --input-from-bundle io.npz
```

```bash Assert outputs match expected
tract model.onnx -O run --input-from-bundle io.npz --assert-output-bundle io.npz
```

```bash Verbose assertion check
tract -v model.onnx -O run --input-from-bundle io.npz --assert-output-bundle io.npz
```

```bash Save outputs to npz
tract model.onnx -O run --input-from-bundle io.npz --save-outputs-npz out.npz
```

</CodeGroup>

With `-v`, a successful assertion prints: `Checked output #0, ok.`

### Input bundles

**`--input-from-bundle <file.npz>`** (also `--input-from-npz`)
Loads tensor *values* from the named `.npz` file. This supersedes the `-i` flag: shapes and dtypes are inferred from the tensors themselves.

**`--input-facts-from-bundle <file.npz>`**
Reads the *shapes and dtypes* from the `.npz` file without loading values. Use this when your model has no embedded type information and you want analysis to succeed before running.

<Info>
  These two flags serve different purposes. `--input-facts-from-bundle` configures the model's input facts during loading; `--input-from-bundle` provides the actual tensor data at runtime. You can combine both.
</Info>

### run options

| Flag | Description |
|---|---|
| `--input-from-bundle <path>` | Load input tensor values from a `.npz` file |
| `--input-from-nnef <dir>` | Load input tensors from a directory of `.dat` files |
| `--assert-output-bundle <path>` | Check outputs against tensors in a `.npz` file |
| `--assert-output <spec>` | Check a single output against a shape spec or file |
| `--save-outputs-npz <path>` | Save model outputs to a `.npz` file |
| `--save-steps <path>` | Save all intermediate tensor values to `.npz` |
| `--dump` | Print output tensor values to stdout |
| `--steps` | Show all intermediate inputs and outputs |
| `-R / --allow-random-input` | Use randomly generated input when none is provided |
| `--approx <level>` | Tolerance for output assertions: `exact`, `close`, `approximate`, `very`, `super`, `ultra` |

## NNEF conversion

Convert a model to NNEF or tract-OPL format for deployment. NNEF files are platform-independent — call `into_optimized()` at runtime on the target device.

```bash
# Convert to a compressed NNEF archive
tract model.onnx -O -i 1,3,224,224,f32 dump --nnef model.nnef.tgz

# Convert to an uncompressed NNEF directory
tract model.onnx -O -i 1,3,224,224,f32 dump --nnef-dir model.nnef/

# Convert to TFLite flatbuffer
tract model.onnx -O -i 1,3,224,224,f32 dump --tflite model.tflite
```

<Note>
  NNEF output encodes the *decluttered* (tract-OPL) form, not the CPU-optimized form. Platform-specific optimizations are applied at load time. This makes NNEF archives portable across architectures.
</Note>

### NNEF extension flags

By default, tract writes standard NNEF. Enable extensions to represent operators that go beyond the NNEF spec:

| Flag | Extension |
|---|---|
| `--opl` | Enable all tract NNEF extensions |
| `--nnef-tract-core` | tract-core operators |
| `--nnef-tract-onnx` | ONNX-specific operators |
| `--nnef-tract-pulse` | Pulsed (streaming) operators |
| `--nnef-tract-extra` | Extra tract operators |
| `--nnef-tract-transformers` | Transformer/LLM operators |

## Input shape syntax

The `-i` flag and several other places accept a shape spec as a comma-separated list of dimensions followed by a dtype suffix:

```
<dim0>,<dim1>,...,<dtype>
```

Examples:

| Spec | Meaning |
|---|---|
| `1,3,224,224,f32` | Batch 1, 3 channels, 224×224 spatial, float32 |
| `1,128,i64` | Batch 1, sequence length 128, int64 |
| `S,f32` | Symbolic batch dimension S, float32 |

Supported dtypes: `f16`, `f32`, `f64`, `i8`, `i16`, `i32`, `i64`, `u8`, `u16`, `u32`, `u64`.

## --transform

Apply a named built-in transformation to the model before running. Transformations run after decluttering and before optimization.

```bash
# Substitute concrete values for symbolic dimensions
tract model.onnx --transform concretize-symbols -i 1,128,i64 dump

# Unfold a KV-cache for attention models
tract model.onnx --transform unfold-kv-cache dump

# Detect and lower transformer patterns (LLMs)
tract model.onnx --transform transformers_detect_all dump
```

<Tip>
  The `--llm` shortcut is equivalent to `--opl --causal-llm-hints -t transformers_detect_all` and is the recommended starting point for working with causal language models.
</Tip>

Multiple transforms can be chained by repeating the flag:

```bash
tract model.onnx \
  --transform concretize-symbols \
  --transform unfold-kv-cache \
  dump
```
