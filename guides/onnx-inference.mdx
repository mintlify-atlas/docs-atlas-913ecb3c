tract supports ONNX operator sets 9 through 18 and passes all real-life ONNX backend integration tests (AlexNet, DenseNet, Inception, ResNet, ShuffleNet, SqueezeNet, VGG, ZFNet).

## Loading and running a model

<Steps>
  <Step title="Add the dependency">
    Add `tract` to your `Cargo.toml`. The `tract` crate bundles ONNX support by default.

    ```toml Cargo.toml
    [dependencies]
    tract = "0.23"
    ```
  </Step>

  <Step title="Load the ONNX file">
    `tract::onnx()` returns an `Onnx` context. Call `.load()` to parse the protobuf file into an `InferenceModel`.

    ```rust
    use tract::prelude::*;

    let inference_model = tract::onnx()?.load("model.onnx")?;
    ```

    Many ONNX models ship with partial or missing shape information. `InferenceModel` accommodates this — tract's type inference fills in the gaps before the model is converted to a fully-typed `Model`.
  </Step>

  <Step title="Set input facts">
    Tell tract the shape and element type of each input. The fact string format is `dim0,dim1,...,dtype` — for example `"1,3,224,224,f32"`.

    ```rust
    use tract::prelude::*;

    let mut inference_model = tract::onnx()?.load("model.onnx")?;
    inference_model.set_input_fact(0, "1,3,224,224,f32")?;
    ```

    Use `InferenceFact::empty()` when you want to leave the type fully unspecified and let tract infer everything from context.
  </Step>

  <Step title="Convert to a typed model">
    `into_model()` runs shape and type inference, producing a `Model` (internally a `TypedModel`) where every tensor dimension and element type is known.

    ```rust
    let model = inference_model.into_model()?;
    ```
  </Step>

  <Step title="Optimize and make runnable">
    `into_runnable()` applies the full optimization pipeline (declutter, kernel selection, layout transforms) and returns a `Runnable` ready for inference.

    ```rust
    let runnable = model.into_runnable()?;
    ```

    The two steps above can be chained:

    ```rust
    let runnable = tract::onnx()?
        .load("model.onnx")?
        .into_model()?
        .into_runnable()?;
    ```
  </Step>

  <Step title="Run inference">
    Pass inputs as ndarray arrays (or anything implementing `IntoInputs`). The result is a `Vec<Tensor>`.

    ```rust
    let output = runnable.run([input_tensor])?;
    let scores = output[0].as_slice::<f32>()?;
    ```
  </Step>
</Steps>

## Complete example: MobileNet v2

The following is adapted from the [`onnx-mobilenet-v2`](https://github.com/sonos/tract/tree/main/examples/onnx-mobilenet-v2) example.

```rust
use anyhow::Result;
use tract::prelude::*;

fn main() -> Result<()> {
    // Load and optimize the model in one chain
    let model = tract::onnx()?
        .load("mobilenetv2-7.onnx")?
        .into_model()?
        .into_runnable()?;

    // Decode and resize the image, then normalize per ImageNet statistics
    let image = image::open("grace_hopper.jpg").unwrap().to_rgb8();
    let resized =
        image::imageops::resize(&image, 224, 224, image::imageops::FilterType::Triangle);

    let input = tract_ndarray::Array4::from_shape_fn((1, 3, 224, 224), |(_, c, y, x)| {
        let mean = [0.485, 0.456, 0.406][c];
        let std  = [0.229, 0.224, 0.225][c];
        (resized[(x as _, y as _)][c] as f32 / 255.0 - mean) / std
    });

    let result = model.run([input])?;

    // Find the top-1 class
    let best = result[0]
        .as_slice::<f32>()?
        .iter()
        .zip(0..)
        .max_by(|a, b| a.0.partial_cmp(b.0).unwrap());
    println!("result: {best:?}");
    Ok(())
}
```

## Python equivalent

<CodeGroup>

```python minimal.py
import tract
import numpy

model = (
    tract.onnx()
    .load("mobilenetv2-7.onnx")
    .into_model()
    .into_runnable()
)

image = numpy.random.rand(1, 3, 224, 224).astype(numpy.float32)
result = model.run([image])
confidences = result[0].to_numpy()
print("top-1 class:", numpy.argmax(confidences))
```

## Handling dynamic shapes

ONNX models often use symbolic batch or sequence dimensions (e.g. `N`, `batch`, `S`). tract represents these as `TDim` — a symbolic dimension that carries an expression rather than a concrete integer.

<Tabs>
  <Tab title="Rust">
    Inspect a symbolic dimension from a `Fact`:

    ```rust
    use tract::prelude::*;

    let model = tract::onnx()?.load("bert.onnx")?.into_model()?;
    let fact = model.input_fact(0)?;
    let seq_dim = fact.dim(1)?; // e.g. "S"
    println!("sequence dim: {seq_dim}"); // prints the symbolic expression

    // Evaluate with a concrete value
    let concrete = seq_dim.eval([("S", 128)])?;
    println!("concrete: {}", concrete.to_int64()?);
    ```

    To fix a symbolic dimension before optimization, use the `ConcretizeSymbols` transform:

    ```rust
    use tract::prelude::*;

    let mut model = tract::onnx()?.load("bert.onnx")?.into_model()?;
    model.transform(ConcretizeSymbols::default().with("S", 128))?;
    let runnable = model.into_runnable()?;
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import tract

    model = tract.onnx().load("bert.onnx").into_model()

    # Fix the symbolic sequence dimension to 128
    model.transform(tract.ConcretizeSymbols({"S": 128}))
    runnable = model.into_runnable()
    ```
  </Tab>
</Tabs>

## Supported ONNX opsets

tract tests against **opset 9** (ONNX 1.4.1) through **opset 18** (ONNX 1.13.0).

Implemented operators include: Abs, Add, AveragePool, BatchNormalization, Cast, Clip, Concat, Conv, ConvTranspose, Einsum, GRU, Gather, Gemm, GlobalAveragePool, LSTM, MatMul, MaxPool, Pad, Relu, Reshape, Resize, Sigmoid, Slice, Softmax, Split, Squeeze, Transpose, Unsqueeze, and [many more](https://github.com/sonos/tract#onnx).

## Common gotchas

<Warning>
  **Tensor Sequences and Optional Tensors are not supported.** tract is designed to flow plain tensors. Operators that produce or consume `Seq<Tensor>` or `Optional<Tensor>` (such as `SequenceInsert`, `SequenceAt`, or optional-typed inputs) will fail to load. This is a structural limitation and unlikely to change.
</Warning>

<Note>
  **Missing operator?** About 85% of ONNX backend tests pass. If a specific operator is unimplemented, check whether it can be fused or replaced at export time (e.g. with `torch.onnx.export` simplification options). You can also open an issue on the tract GitHub — many operators are straightforward to add.
</Note>

<Tip>
  For models with complex preprocessing baked in (e.g. `Resize` with non-trivial coordinate transforms), consider moving those steps outside the model and handling them in application code.
</Tip>
